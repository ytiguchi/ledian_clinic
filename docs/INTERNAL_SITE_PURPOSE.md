# 内部サイト（Internal Site）の目的と方針

## 目的

1. **メニュー・料金管理**: 施術メニューと料金プランの管理
2. **キャンペーン管理**: キャンペーン情報の管理
3. **症例写真管理**: ビフォー・アフター写真の管理
4. **トレーニング**: スタッフが新しい情報をまとめて確認できる場所

## 利用者

- **院長**: 経営判断、承認
- **エンジニア**: システム管理、データ管理
- **スタッフ全員**: 約30名（日常的な情報確認、トレーニング）

## 将来の展開

### データ連携

内部サイトを**データソース（DB）**として使用し、以下に反映：

1. **公開WEBサイト**: 内部サイトで管理したデータを公開サイトに反映
2. **POSシステム**: 料金・メニュー情報をPOSに連携
3. **カルテシステム**: 施術情報をカルテシステムに連携
4. **その他システム**: 必要に応じて追加

### 連携方式

- API経由でのデータ提供
- 定期的な同期処理
- リアルタイム更新（必要な場合）

## 運用頻度

- **日次**: 毎日のデータ更新・確認

## 現在の実装状況

### ✅ 実装済み

- 料金管理機能（CRUD完備）
- キャンペーン管理機能
- 症例写真管理機能
- 基本的なデータ構造

### 🚧 今後実装が必要

1. **トレーニング機能**
   - 新着情報のまとめ表示
   - スタッフ向けの教育コンテンツ
   - FAQ・よくある質問
   - 施術方法の説明資料

2. **データエクスポート機能**
   - CSV/PDF/JSON出力
   - POSシステム向けフォーマット
   - カルテシステム向けフォーマット

3. **APIエンドポイント**
   - 公開WEBサイト向けAPI
   - POSシステム向けAPI
   - カルテシステム向けAPI

4. **承認フロー**
   - 院長承認機能
   - 変更履歴管理

5. **権限管理**
   - 役割別アクセス制御
   - 編集権限の管理

## 優先順位

### Phase 1（現在〜短期）
- トレーニング機能の追加
- データエクスポート機能（CSV/PDF/JSON）

### Phase 2（中期）
- APIエンドポイントの整備
- 公開WEBサイトとの連携

### Phase 3（長期）
- POSシステム連携
- カルテシステム連携
- 承認フロー
- 権限管理の強化

## 技術的な考慮事項

- **データ構造**: 将来的な連携を見据えた設計
- **API設計**: RESTful APIの標準化
- **認証・認可**: Cloudflare Access + 役割ベースアクセス制御
- **データ同期**: バッチ処理 vs リアルタイム更新の検討
